# Principles

## Core Values

### 1. 推測ヒアリングによる要求引き出し
エージェントは常に「推測 + 質問」の形式でユーザーと対話し、暗黙の要求を明示化する。
空白の質問は避け、仮説を提示することでユーザーの思考を促す。

### 2. エージェントの自律性
SPEC確定後は、エージェントが最適な順序・方法で実装を進める。
タスクを上から順にこなす必要はなく、依存関係や効率を考慮して自律的に判断する。

### 3. SPEC as an Asset
すべてのSPECは知識資産として蓄積される。
特にnotes.mdは「発見、試行錯誤、予期せぬ解決策」のストックとなり、将来の類似課題で再利用される。

### 4. シンプルに始められ、カスタマイズ可能
最小限のセットアップで開始でき、必要に応じてカスタマイズできる。
複雑さを強制せず、プロジェクトの成長に合わせて拡張可能。

## Decision Framework

### 意思決定プロセス

```
1. 推測ヒアリング（Q-SPEC Framework）
   ↓
   ユーザーの意図を引き出し、要求を明確化
   
2. SPEC確定
   ↓
   requirements.md, design.md, tasks.mdを作成
   
3. エージェント主導の実装
   ↓
   最適な順序・方法で自律的に実装
   
4. ユーザー介入（必要時）
   ↓
   ユーザーはいつでも介入可能だが、基本はエージェント主導
```

### ヒアリングの重要性

良いプロダクトは良いヒアリングから生まれる。
エージェントは以下を意識してヒアリングを行う：

- **S (Scope)**: 全体像と粒度の把握
- **P (Pathfinding)**: 最適なヒアリング経路の探索
- **E (Elicit)**: 具体例による暗黙知の引き出し
- **C (Constrain)**: 制約による実現可能な範囲への収束

## Trade-off Priorities

### エージェントの自律性 > ユーザーの細かい制御
エージェントに最大限の自律性を与え、ユーザーは方針レベルの指示に留める。
細かい実装手順はエージェントの判断に委ねる。

### シンプルさ > 機能の豊富さ
複雑な機能よりも、シンプルで理解しやすい構造を優先する。
ただし、必要なカスタマイズ性は確保する。

### 規約の一貫性 > 柔軟性
- **SPEC構造**: 固定（requirements.md, design.md, tasks.md, notes.md）
- **Steering構造**: 基本固定（vision.md, principles.md, tech.md, constraints.md）+ 追加可能

一貫性により、エージェントとユーザーの認知負荷を下げる。

## Consistency Rules

### SPEC管理
- SPECは `.kiro/specs/YYYYMMDD-HHMM_spec-name/` 形式で管理
- 4つのファイル構造は固定（追加・削除不可）
- notes.mdは追記のみ（既存内容の編集・削除禁止）

### Steering管理
- Steeringは不変の本質的方針のみを記述
- プロジェクト進行で陳腐化する内容は含めない
- 技術的詳細や可変情報はSPECに記述

### コンテキスト管理
- コンテキストファイルはなるべく肥大化させない
- 重複した情報を複数ファイルに記述しない
- エージェントのトークン制限を常に考慮

## Feature Addition Criteria

新機能やルールを追加する際の判断基準：

### 1. コアバリューに直結するか
推測ヒアリング、エージェント自律性、SPEC as an Assetなど、プロジェクトの本質的価値に貢献するか。

### 2. シンプルさを維持できるか
追加することで複雑さが増さないか。
学習コストや認知負荷が許容範囲か。

### 3. 実用性があるか
実際のプロジェクトで必要とされているか。
理論的な完璧さよりも、実用的な価値を優先。

### 4. エージェントの自律性を妨げないか
エージェントの判断を過度に制約しないか。
柔軟な実装を可能にするか。

**すべての基準を満たす場合のみ、追加を検討する。**
